# Seed scripts

Synthetic-data loaders for the dev environment. Each backend service has its
own loader at `services/<svc>/seed/seed.py`; this directory holds the shared
primitives those loaders import.

> **PHI / synthetic data disclaimer.** Every seeded row is fabricated. Names,
> MRNs, NPIs, addresses, and emails are generated by deterministic functions
> in `_common.py` — none are intended to match real individuals or
> organizations. Don't run these against a database that contains real PHI.

## Why a single shared module?

The six services own separate Lakebase databases and never query each other.
But a "patient summary" demo only feels real if `appointment.patient_id`
resolves to a `patient.id` row that actually exists, and `lab_order.patient_id`
points at the same person. We achieve that without coupling the services by
deriving every cross-service ID from `(entity, ordinal)` via UUIDv5 with a
fixed namespace:

```python
ID.patient(0)   # always a3c1... whatever process or DB you run it from
ID.provider(2)  # always 7e15...
```

So `appointment-svc/seed/seed.py` writes `patient_id = ID.patient(i %
NUM_PATIENTS)` and `patient-svc/seed/seed.py` writes a row with that exact
UUID — no cross-DB lookup, no shared state at runtime, just a fixed
algorithm.

## Idempotency

Every seed uses `INSERT ... ON CONFLICT (id) DO NOTHING` (or the natural-key
equivalent for catalog tables). Running the same seed twice is a no-op.

## Volume knobs

`_common.py` exposes `NUM_PATIENTS`, `NUM_APPOINTMENTS`, etc. as module
constants seeded from env vars (`SEED_NUM_PATIENTS=10` for a smoke run, etc.).
Defaults are the demo-friendly counts: 50 patients, 5 providers, 200
appointments, 60 lab orders, 30 prescriptions, 10 refills, 100 invoices.

## Running

From the repo root, with `.env` populated for each service:

```bash
make seed-dev                        # all six, in dependency order
make seed-patient                    # one at a time
SEED_NUM_PATIENTS=5 make seed-patient  # small smoke run
```

Each Make target sources `services/<svc>/.env` so the loader sees the same
`PGHOST` / `PGUSER` / `ENDPOINT_NAME` the runtime app does. The loader mints
its own Lakebase OAuth credential from your CLI token; rows therefore land
with an audit trail (`created_by` = your email or
`SEED_OPERATOR_EMAIL` if set).

## Layout

```
scripts/seeds/
├── _common.py     # shared ID factory, connect(), generators
└── README.md      # this file

services/<svc>/seed/
└── seed.py        # per-service loader; thin wrapper around _common.py
```

The seed scripts deliberately do not depend on each service's runtime package
(no import of `patient.app` etc.). They run against the database directly,
so a schema change that breaks the runtime code doesn't block seeding.
